# CHAPTER 1 ASSIST

CHAPTER 1 ASSIST is an async ERP and accounting AI assistant built with **LangGraph**, **LangChain**, **ChromaDB**, **FastAPI**, and **Ollama**.

The assistant accepts natural-language ERP queries, retrieves the most relevant ERP tools using semantic search, calls only the required tools, and returns a deterministic JSON response from structured Chapter 1 ERP/API data.

---

## Overview

Most ERP assistants expose every available tool to the LLM on every request. This project follows a cleaner workflow:

1. Understand the user query.
2. Retrieve only the most relevant ERP tool documentation using ChromaDB.
3. Bind only the selected tools to the LLM.
4. Let the LLM perform tool-calling only.
5. Build the final JSON response using Python instead of making another LLM call.

This reduces unnecessary tool exposure and keeps the final response more predictable.

## Features

- Async LangGraph workflow
- FastAPI backend with a `/chat` endpoint
- Semantic tool retrieval using ChromaDB
- Local LLM support through Ollama
- Tool-calling with LangChain tools
- Deterministic final JSON response builder
- Step-level timing logs for graph debugging

## Current Supported ERP Tools

The active API tool layer currently supports:

| Tool | Purpose | API Endpoint |
|---|---|---|
| `get_product_list` | Fetch products, inventory, item details, HSN, GST rate, stock, SKU, UOM | `/productList` |
| `get_purchase_list` | Fetch purchase invoices, supplier bills, vendor bills, purchase amounts, outstanding purchases | `/purchaseList` |
| `get_sales_list` | Fetch sales invoices, customer bills, sales amounts, outstanding sales | `/salesList` |

## How It Works

```text
User Query
   |
   v
Semantic Search Node
   |
   v
Retriever searches ChromaDB tool documents
   |
   v
Chat Model Node
   |
   v
LLM receives only relevant tools
   |
   v
Routing Node
   |
   v
Tool Node executes API tools
   |
   v
Deterministic Final Node
   |
   v
Final JSON Response
```

---

## LangGraph Flow

The graph contains the following main nodes:

| Node | Responsibility |
|---|---|
| `semantic_search` | Finds the most relevant tools for the query |
| `chat_model` | Uses the LLM only to decide tool calls |
| `tools` | Executes selected ERP tools |
| `deterministic_final` | Merges tool outputs into final JSON |
| `routing_node` | Routes execution depending on whether tool calls exist |

The final response is generated by Python, not by a second LLM call.

### Chat Endpoint

```http
POST /chat
```

Request body:

```json

  "query": "Show me all purchase invoices from Maharashtra."
```

```bash
curl -X POST "http://127.0.0.1:8000/chat" \
```

Example response format:

```json
  "timings": [
    {
      "node": "semantic_search",
      "duration_sec": 0.12
```

## Setup Instructions

### 1. Clone the Repository

```bash
git clone <your-repository-url>
cd CHAPTER-1-ASSIST
```

---

### 2. Create a Virtual Environment

```bash
python3 -m venv venv
source venv/bin/activate
```

For Windows:

```bash
python -m venv venv
venv\Scripts\activate
```

---

### 3. Install Dependencies

```bash
pip install -r requirements.txt
```

---

### 4. Install and Run Ollama

Install Ollama from the official website, then pull the required models:

```bash
ollama pull qwen3:14b-q4_K_M
ollama pull nomic-embed-text
```

Make sure Ollama is running:

```bash
ollama serve
```

---

### 5. Configure Environment Variables

Create a `.env` file or export variables in your terminal:

```bash
export CHP1_API_BASE_URL="https://dev.chapter1.finance/ai/"
export CHP1_API_TIMEOUT="30"
export COMPANY_ID="355"
```

Optional:

```bash
export CHP1_API_TOKEN="your-api-token"
```

> Security note: Do not commit API tokens or secret authorization values to GitHub. Keep them in environment variables.

---

### 6. Run the App

```bash
python3 fast_main.py
```

Then open:

```text
http://127.0.0.1:8000
```

## Filtering and Field Projection

Tools support dynamic filtering and selected output columns.

### Field Projection

The assistant can request only selected fields:

```json
{
  "fields": ["invoiceNo", "billToName", "netAmount"]
}
```

### Filters

Filters are passed as structured JSON:

```json
{
  "filters": {
    "billToState": "Maharashtra"
  }
}
```

Supported filter operators:

| Operator | Meaning |
|---|---|
| `eq` | Equal to |
| `contains` | Text contains |
| `in` | Matches one value from a list |
| `gt` | Greater than |
| `gte` | Greater than or equal to |
| `lt` | Less than |
| `lte` | Less than or equal to |

Example:

```json
{
  "filters": {
    "outstanding": {
      "gt": 10000
    }
  }
}
```

## Response Status Values

The deterministic final node can return:

| Status | Meaning |
|---|---|
| `success` | Records were found successfully |
| `partial_success` | Some tools returned data while others returned empty results or errors |
| `no_matching_records` | Tools ran but no matching records were found |
| `error` | Tool output or API execution failed |
| `graph_timeout` | Graph execution exceeded timeout |
| `graph_error` | Unexpected graph execution error |

---

## Why Deterministic Final JSON?

Instead of asking the LLM to produce the final answer after tool execution, this project uses Python to merge tool outputs.

Benefits:

- Reduces hallucination risk
- Avoids invalid JSON from the LLM
- Removes one extra LLM call
- Makes API responses predictable
- Makes errors easier to debug
